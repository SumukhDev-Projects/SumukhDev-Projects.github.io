Financial Advisor RAG

Full-stack RAG system for financial advisors to query client PDFs in natural language — grounded answers with citations, portfolio summaries, and drafted client emails. Audit logging for compliance.

FastAPIClaude APIPostgreSQLpgvectorsentence-transformersReactDocker
The Problem

Financial advisors spend significant time manually digging through client PDFs before every meeting. The goal was a system where an advisor could ask "What's John's current equity allocation?" and get an answer in seconds, grounded in the actual document — not an LLM guess.

Architecture
Upload: pypdf + LangChain splitters (chunk=800, overlap=150) → sentence-transformers/all-MiniLM-L6-v2 (local) → pgvector storage in PostgreSQL Query: embed question → cosine similarity → top-5 chunks → Claude receives chunks + question → Grounded answer with source citations Audit: every query + response + retrieved chunks logged with timestamp + advisor ID for compliance trail
Key Decisions
  • Local embeddings: sentence-transformers means document content never leaves the infrastructure during indexing. Zero per-request embedding cost.
  • pgvector over Pinecone: Keeping vector search in the same Postgres DB simplified the stack. At this scale, pgvector with HNSW indexing was fast enough.
  • Audit logging from day one: Financial advisors operate under compliance requirements. Every query and source reference writes to audit_logs — a core feature, not an afterthought.
What I Learned

Chunk size and overlap have a bigger impact on retrieval quality than the choice of LLM. 800 token chunks with 150 token overlap hit the right balance for financial documents — specific enough for data points, long enough to preserve context. The hardest part was making the system say "I don't know" cleanly when the answer wasn't in the documents.

NL-SQL Analyst

Natural language to SQL — ask a question, Claude generates the SELECT, backend validates and executes it, frontend auto-renders results as table or chart. Two-layer read-only enforcement; 31 unit tests.

FastAPIClaude APISQLitepandasReactPlotly
The Problem

Most analysts spend disproportionate time writing boilerplate SQL for straightforward lookups. The challenge: can an LLM close that gap safely — where safely means the system can't mutate or delete data regardless of what the user asks or what Claude generates?

The Safety Layer
Layer 1 — Connection level: sqlite3.connect(f"file:{db}?mode=ro", uri=True) PRAGMA query_only = ON Layer 2 — Statement whitelist: ✓ SELECT, WITH (CTEs) ✗ DROP, DELETE, INSERT, UPDATE, CREATE, ALTER, TRUNCATE, ATTACH, semicolons

The worst outcome is a slow SELECT. The database cannot be modified under any circumstances — even if Claude generates something unexpected.

Testing

Shipped 31 unit tests across SQL safety, schema correctness, query execution, and chart generation — all against the real SQLite database, not mocks.

SQL SafetySchema TestsExecution TestsChart Gen
What I'd Do Differently

The full schema goes to Claude on every request — fine for a small DB, won't scale to hundreds of tables. Next iteration: embed table/column descriptions and retrieve only schema relevant to the user's question, keeping the prompt tight.

Emotions AI

Real-time emotion detection from facial expressions and text sentiment. TypeScript frontend built for low-latency interaction with confidence score outputs across emotion categories.

TypeScriptReactAI InferenceSentiment Analysis
What It Does

Classifies emotional state in real time from two modalities: facial expressions via webcam and text input via sentiment analysis. Returns a primary emotion label plus a confidence distribution — not just what emotion is detected, but how certain the model is.

Why TypeScript

Latency between capture and result has to feel instant. TypeScript gave strict typing across the inference response shape, catching subtle bugs early where confidence score keys didn't match expected emotion labels. For AI apps where the output schema matters, typed frontends pay for themselves quickly.

Interesting Challenges
  • Confidence thresholding: Ambiguous faces spread probability across multiple emotions. Had to decide at what threshold to show "uncertain" rather than commit to a label.
  • Async inference loop: Keeping the capture-infer-render loop smooth without dropping frames required careful async handling — inference can't block the UI thread.
  • Modality disagreement: When typed text and facial expression express different emotions, both are surfaced independently rather than reconciled.

How I Cut ETL Review Time by 70% Without Rewriting Everything
The problem wasn't the pipeline — it was that nobody knew when it broke. Here's the validation layer that changed that.
5 min read · ETL, Python, Data Quality

The real problem wasn't the pipeline

When I joined UIUC Extension, there was an existing process for collecting public health data from field sources — spreadsheets, form submissions, manual exports. It more or less worked. The problem was that when it didn't, no one knew until someone downstream needed a number that wasn't there. A null in a USDA funding report is a different kind of problem than a null in a sales dashboard.

The approach: validate at the row level, loudly

The fix wasn't a new pipeline — it was adding row-level validation to the existing ingestion using Great Expectations. Three categories: null checks on required fields, range checks on numeric values (a health metric outside expected bounds is almost always a data entry error), and duplicate detection on composite keys.

The key was making failures specific. Instead of logging "validation failed," the system writes the exact row, column, and rule that failed to a separate validation log table. When the morning run has issues, the message is "rows 47, 89, 203 failed range check on bmi_value" — not "something went wrong."

What actually drove the 70% reduction

Before: a reviewer would spend 2–3 hours per run manually spot-checking outputs against source files. After: validation failures surface automatically, and the reviewer triages a structured exception list. On clean runs — which became the majority — review time dropped to under 15 minutes.

The secondary benefit: field data collectors started receiving automated feedback when their submissions had issues. Data quality at the source improved over time because the feedback was specific enough to act on.

The lesson

Pipelines that run silently are the most dangerous kind. Validation infrastructure pays back every time a bad data point doesn't make it into a report someone will make a decision from.

RAG vs Fine-Tuning: What Building a Financial Document Assistant Actually Taught Me
Everyone has a take on this. Here's what building the thing for real taught me about when each approach makes sense.
6 min read · RAG, LLMs, Architecture

The question everyone asks wrong

"RAG or fine-tuning?" is usually the wrong frame. Better: what kind of knowledge does the model need, and how does that knowledge change? For the Financial Advisor RAG system, the answer was obvious — the knowledge was client-specific, changed constantly, and couldn't be baked into weights. Fine-tuning was never on the table.

Where RAG breaks down

RAG fails in two predictable places: retrieval quality and context window composition. I tested five chunk sizes on financial documents. Too small and you retrieve fragments without enough context. Too large and your top-k results eat the context window with irrelevant surrounding text. For financial PDFs, 800 tokens with 150-token overlap was the sweet spot — but this is domain-specific and you have to test it.

Context window composition is subtler. Chunks retrieved by cosine similarity aren't necessarily ordered by relevance to the specific question. Re-ranking retrieved chunks before passing them to Claude showed measurable improvement on multi-part questions.

The compliance angle nobody talks about

Financial applications need an audit trail. When an advisor tells a client something based on their data, there needs to be a record of what the system used to arrive at that answer. Audit logging — every query, every retrieved chunk, every response — isn't a nice-to-have. It's what makes the system usable in a regulated context. Build this from day one.

When fine-tuning actually makes sense

Fine-tuning teaches a model style and format, not facts. If you want consistent structured outputs, the right professional register, or firm-specific abbreviations — that's fine-tuning territory. Combine it with RAG for factual grounding and you have a system that sounds right and is right.

The Dashboard Nobody Opens Is a Failure, Not a Deliverable
I've built dashboards that got used and ones that didn't. The difference wasn't the tool or the design. It was something earlier in the process.
4 min read · Dashboards, Stakeholders, BI

A dashboard is not a deliverable

At Holiday Channel, I built real-time Power BI dashboards tracking CAC, LTV, and marketplace health. They looked good. The data was accurate. For two weeks, the right people opened them before strategy meetings. I've also seen dashboards that get built, get shared, and quietly stop being opened — not because the data is wrong, but because nobody remembers to look at them when making a decision.

The question to ask before you build anything

"What decision will this change, and who makes that decision?" If you can't answer that specifically, you're probably building something that won't get used. "Better visibility" is not a decision. "The growth team deciding whether to run another acquisition campaign this month" is — and it tells you exactly what metrics matter, what time horizon to show, and who needs to be in the room.

The practical fix

Before starting any dashboard project, I now run one conversation covering: what decision this connects to, how often that decision gets made, and what they're currently doing instead. That last one is the most revealing. If they're already emailing a weekly CSV around, your dashboard needs to replace that exact workflow — not add another place to check.

At UIUC Extension, the dashboards that get used consistently are tied directly to the USDA reporting cycle. The decision is concrete, the timing is fixed, and the users had a painful manual process before. The dashboard removed the pain. That's why it gets opened.

On design

Dashboard design matters less than dashboard placement. A mediocre dashboard bookmarked on the right person's browser and opened every Monday drives more decisions than a beautiful dashboard they have to remember exists. Get the workflow right first. Then make it beautiful.
